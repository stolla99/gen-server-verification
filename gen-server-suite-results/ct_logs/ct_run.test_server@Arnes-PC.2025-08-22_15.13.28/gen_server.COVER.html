22 -module(gen_server).
23 -moduledoc """
24 Generic server behavior.
25
26 This behavior module provides the server in a client-server relation.
27 A generic server process (`gen_server`) implemented using this module
28 has a standard set of interface functions and includes functionality
29 for tracing and error reporting. It also fits into
30 an OTP supervision tree. For more information, see section
31 [gen_server Behaviour](`e:system:gen_server_concepts.md`)
32 in OTP Design Principles.
33
34 A `gen_server` process assumes all specific parts to be located
35 in a callback module exporting a predefined set of functions.
36 The relationship between the behavior functions
37 and the callback functions is as follows:
38
39 ```text
40 gen_server module Callback module
41 ----------------- ---------------
42 gen_server:start
43 gen_server:start_monitor
44 gen_server:start_link -----> Module:init/1
45
46 gen_server:stop -----> Module:terminate/2
47
48 gen_server:call
49 gen_server:send_request
50 gen_server:multi_call -----> Module:handle_call/3
51
52 gen_server:cast
53 gen_server:abcast -----> Module:handle_cast/2
54
55 - -----> Module:handle_info/2
56
57 - -----> Module:handle_continue/2
58
59 - -----> Module:terminate/2
60
61 - -----> Module:code_change/3
62 ```
63
64 If a callback function fails or returns a bad value,
65 the `gen_server` process terminates. However, an exception of class
66 [`throw`](`erlang:throw/1`) is not regarded as an error
67 but as a valid return, from all callback functions.
68
69 A `gen_server` process handles system messages as described in `m:sys`.
70 The `m:sys` module can be used for debugging a `gen_server` process.
71
72 Notice that a `gen_server` process does not trap exit signals
73 automatically, this must be explicitly initiated in the callback module.
74
75 Unless otherwise stated, all functions in this module fail
76 if the specified `gen_server` process does not exist
77 or if bad arguments are specified.
78
79 The `gen_server` process can go into hibernation (see `erlang:hibernate/3`)
80 if a callback function specifies `'hibernate'` instead of a time-out value.
81 This can be useful if the server is expected to be idle for a long time.
82 However, use this feature with care, as hibernation implies at least
83 two garbage collections (when hibernating and shortly after waking up)
84 and is not something you want to do between each call to a busy server.
85
86 If the `gen_server` process needs to perform an action after
87 initialization or to break the execution of a callback into multiple steps,
88 it can return `{continue, Continue}` in place of
89 the time-out or hibernation value, which will invoke
90 the [`Module:handle_continue/2`](`c:handle_continue/2`) callback,
91 before receiving any external message / request.
92
93 If the `gen_server` process terminates, e.g. as a result of a function
94 in the callback module returning `{stop,Reason,NewState}`,
95 an exit signal with this `Reason` is sent to linked processes and ports.
96 See [Processes](`e:system:ref_man_processes.md#errors`)
97 in the Reference Manual for details regarding error handling
98 using exit signals.
99
100 > #### Note {: .info }
101 >
102 > For some important information about distributed signals, see the
103 > [_Blocking Signaling Over Distribution_][1]
104 > section in the _Processes_ chapter of the _Erlang Reference Manual_.
105 > Blocking signaling can, for example, cause call time-outs
106 > in `gen_server` to be significantly delayed.
107
108 [1]: `e:system:ref_man_processes.md#blocking-signaling-over-distribution`

446

448 when it is about to terminate.
449
450 It is to be the opposite of [`Module:init/1`](`c:init/1`)
